Implementation Notes — KSP 1 Meme Mutation Radar (Frontend)

Running log for the Meme Mutation Radar detail-panel UI. Captures every styling, state, and component decision not spelled out in the spec. Component source carries no comments per CLAUDE.md §3.1; all such commentary lives here and in implementation-notes.md.

Frontend UI & Component Refinements

  • Styling, charts & state hooks. The radar panel is a self-contained frontend/src/components/MutationRadar.jsx composed of four semantic sub-components — TrendVelocityBadge, DriftVectorGauge, VelocityGraph/SmallSampleNotice, and EvolutionTimeline. No charting dependency was added: the drift gauge and velocity graph are pure CSS meters (gradient track + absolutely-positioned thumb/threshold marker), and the evolution "vector cluster" is a hand-rolled inline <svg> scatter — keeping the bundle untouched (package.json unchanged) per the "lightweight" directive. Styling is scoped to .radar-* classes in MutationRadar.css and reuses the existing theme tokens (--bg, --fg, --muted, --accent, --card) so the panel is visually native to the app. State is a single useState mutation index hydrated by one useEffect fetch of /mutations on mount; the per-meme radar model is derived (not stored) from the selected hit via toMutationRadarModel(active, mutationIndex), so no extra re-render churn and no stale duplication of telemetry. The API client layer (api.js) owns all payload mapping: fetchMutations, buildMutationIndex, and toMutationRadarModel normalize template_drift_score, trending_mutation, and lineage_cache (accepting both the spec name lineage_cache and the live backend field lineage).
  • Small-sample conditional rendering. The guardrail is data-driven, not a magic number in the view. buildMutationIndex reads min_members from the /mutations response (default 5), and toMutationRadarModel sets accumulatingBaseline when the matched template's accumulating_baseline flag is true or member_count < min_members (this is the velocity == 0.0 guardrail surfaced by the backend for N < 5). The container then renders a strict either/or: accumulatingBaseline ? <SmallSampleNotice/> : <VelocityGraph/> — the live velocity graph is replaced, never merely hidden, satisfying the spec. The banner copy is the exact mandated string ("Accumulating baseline data: This template requires at least 5 instances for drift telemetry"), with the threshold count interpolated from minMembers so it stays correct if the backend constant changes. This path is asserted by TC-UI-007 (see ID-collision note below).
  • Responsive layout & integration. The radar mounts as a third <aside> column inside the existing meme-detail modal (.modal-inner), which became flex-wrap: wrap with a raised max-width (1000→1240px) and a vertical scroll cap (max-height: 88vh). The image is now flex: 0 0 auto (≤440px), the text column flex: 1 1 280px, and the panel flex: 1 1 300px (≤360px), so on wide viewports it reads as a side panel and on tablet widths it wraps gracefully. A @media (max-width: 900px) query collapses the modal to a single stacked column with a full-width image, and the panel drops its max-width. The stale .lineage block (and its CSS rule) was retired because the EvolutionTimeline now renders template + variants, avoiding duplicate lineage UI.
  • Graceful degradation (degradable-architecture rule). Every external read fails soft. A failed/initial /mutations fetch resolves to buildMutationIndex(null) (empty index, ok: false); an unmatched template yields hasTelemetry: false. In that state the badge shows "Stable Format", the drift gauge shows "Drift telemetry unavailable" (the standard null-lineage value), the velocity slot shows "Mutation telemetry unavailable", and the timeline falls back to the hit's own template/variants or "No lineage recorded". The panel never throws on missing data, mirroring the backend's 200-with-null contract (CLAUDE.md §2.3 invariant 5).
  • Drift-gauge scale decision. The spec labels the gauge 0.0 (Canonical) → 1.0 (Extreme Drift), but the backend cosine template_drift_score is mathematically bounded to [0, 2] (see implementation-notes.md). The gauge therefore renders the spec's 0–1 labeled scale and clamps the fill/thumb to [0, 1] while displaying the raw numeric value, plus a three-zone readout (Canonical / Drifting / High-Irony Mutation). Because the current /search MemeHit does not yet surface per-meme template_drift_score, the client maps it defensively and shows the null state until the backend wires it through — no fabricated value.
  • Velocity graph as a scalar meter. The /mutations payload exposes a single velocity scalar (centroid displacement), not a time series, so the "live velocity graph" is an honest horizontal meter scaled against the trending threshold (with a threshold tick and a "hot" accent state above it) rather than a fake sparkline. TC-UI-008 asserts the badge state derived from trending_mutation.
  • Test-ID collision (TC-UI-001/002 → TC-UI-007/008). The spec requested test IDs TC-UI-001 and TC-UI-002, but both already exist in TESTS.md (cold-start search and grid responsiveness) and are referenced by the traceability matrix. Following the precedent already set for TC-VEC-002→TC-VEC-003 and TC-FAIL-007→TC-FAIL-009 in implementation-notes.md, the new cases land at the next free IDs — TC-UI-007 (accumulating-baseline banner) and TC-UI-008 (Trend Velocity badge) — with the coverage dashboard and matrix updated accordingly.
  • Scatter coordinates are illustrative. The API exposes variant names only, not per-variant vectors, so the mini scatter places the template at a fixed core and distributes variants deterministically around a ring by index. It is a legible "vector cluster" affordance, not a projection of real embeddings — documented here so it is not mistaken for true 2-D drift coordinates.
